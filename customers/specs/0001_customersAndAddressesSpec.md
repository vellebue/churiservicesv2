# Customers And Addresses

This spec will define basic operations to handle customers and its addresses

## Entities

To handle customers and their addresses it will be defined these entities:

Address type entity to define different types of addresses (read only entity):

- addressType: String 10 chars and primary key.
- description: Address description 255 chars and required.
- addressKey: i18n key to keep localized description.

This entity table will be populated with these values:

| addressType | description              | addressKey                 |
|-------------|--------------------------|----------------------------|
| COMMERCIAL  | Commercial Address       | address.type.commercial    | 
| SOCIAL      | Social Address           | address.type.social        |
| BILLING     | Billing Address          | address.type.billing       |
| LOCAL       | Local delegation address | address.type.local         |


Address entity to store in table ADDRESSES

- addressId: Long autogenerated id with sequence SEQ_ADDRESSES and primary key.
- addressType: Required, it references Address Type primary key.
- name: String 1024 chars required, the name of the owner for this address (person or organization).
- address: String 1024 chars required, describes specific local address (street and building number and so on).
- postalCode: String 10 chars required, it is the local postal code.
- city: String 50 chars required, the name of the city where this address applies.
- country: String 2 chars required, it is the country ISO Code that applies for this address.
- region: String 10 chars required, it is the region code that applies to this address.

Customer entity to store in table CUSTOMERS

- customerId: Long autogenerated id with sequence SEQ_CUSTOMERS and primary key.
- commercialName: String 512 chars required, it is the common name of the customer.
- socialName: String 1024 chars required, it is the social (or legal) name of the customer.
- vatNumber: String 20 chars required, it is the VAT or TAX identifier depending on the country.
- customerAddress: Required, Reference to **Address entity** representing main reference address for this customer. Qualify addressType as COMMERCIAL in this case.
- socialAddress: Required, Reference to **Address entity** representing the address for legal and TAX commitments. Qualify addressType as SOCIAL in this case.
- billingAddress: Optional, Reference to **Address entity** representing the address to receive invoices or other legal documents. If not present social address will replace this purpose. Qualify addressType as BILLING in this case.


Customer Delegation entity to store in table CUSTOMER_DELEGATIONS:

- delegationId: Long primary key generated by SEQ_CUSTOMER_DELEGATIONS, it is the own delegation identifier.
- customerId: Long foreign key, it is the reference to Customer entity this delegation belongs to.
- orderId: Integer required, this is the sequence order of this delegation in customer (from 0 to N but not required consecutive numbers). If not notified when creating it will be filled with the highest number for customer plus 1. If not notified when updating it will be kept at its current value. Anyway if notified and if this number matches an existing previous orderId for the same customerId notify as an error "orderId X is already registered for customer $customerId".
- name: String 512 chars required, it is the name for this client delegation.
- address: Required, reference to the **Address entity** that represents the main address for this customer delegation. Qualify addressType as LOCAL in this case.
- billingAddress: Optional, Reference to the **Address entity** that represents a particular billing address for this delegation. Qualify addressType as BILLING in this case.

# Constraints and considerations

- Each address belongs to a certain entity/delegation so consider it when removing entities and delegations. Cascade removal.
- VAT Numbers should be ensured to be unique per country. Notice a validation error in case of reuse a VAT number for the same country.
- All filtered GET (list) operations defined in this spec (currently the customers list and the delegations list) MUST return paged results using the common Pagination contract described in `agent.md`. Single-resource GETs (get by id) are NOT paged. Each filtered use case below only declares its allowed sort fields, default sort, and tie-breaker; the request parameters (`offset`, `limit`, `sort`), the response envelope (`content`, `offset`, `limit`, `hasMore`, `sort`), the defaults, the limits, and the validation rules all come from the common pattern and MUST NOT be redefined here.

# Use Cases

These use cases should be developed:

## Customers

- Create A Customer (with no delegations).
- Update a Customer (keep its delegations untouched). Full update no upsert.
- Delete a Customer and its depending entities.
- Get a customer by its id retrieving its depending addresses and delegations.
- Get a filtered list of customers by:
    - Commercial Name: Using '*' as wildcard.
    - Social Name: Using '*' as wildcard.
    - VAT Number: Using '*' as wildcard.

  When retrieving result give back Customer fields and populated addresses. No delegation details are required.

  Result is paged per the **Pagination** rules in `agent.md`.
    - Allowed sort fields: `commercialName`, `socialName`, `vatNumber`, `customerId`.
    - Default sort: `commercialName,asc`.
    - Tie-breaker: `customerId`.

## Delegations

- Create (add) a delegation to a previous existing customer.
- Update a delegation of a customer. Full update no upsert.
- Delete a delegation of a customer.
- Get a delegation by its delegation id retrieving all data.
- Get a filtered list of delegations by:
    - name: Using '*' as wildcard.
  When retrieving give back all data for each delegation.

  Result is paged per the **Pagination** rules in `agent.md`.
    - Allowed sort fields: `name`, `orderId`, `delegationId`.
    - Default sort: `name,asc`.
    - Tie-breaker: `delegationId`.
